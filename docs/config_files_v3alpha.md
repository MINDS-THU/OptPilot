# OptPilot V3 Alpha Config Files

This document defines the user-facing configuration files for OptPilot v3 alpha.

The goal is to make studies easy to describe without exposing every internal execution detail. Users should be able to say:

- what environment should be evaluated
- what kind of candidate the optimization method must produce
- how the candidate is run in the environment
- where observations and metrics come from
- which optimization method should receive observations and propose new candidates
- what objective and budget define the study

OptPilot then expands these small user-facing files into the internal canonical `StudySpec` used by the runner, evidence store, artifact materializer, validator, scheduler, and backend.

## 1. Design Principles

The config design follows these rules.

1. Keep only necessary fields.
2. Keep user-facing concepts stable and small.
3. Make every allowed enum value explicit.
4. Use `config` only as an intentional escape hatch for user-owned implementations.
5. Do not expose internal adapter names as user-facing concepts unless they describe a real user decision.
6. Let OptPilot infer execution, evidence, materialization, validation, and reproducibility defaults.
7. Store large code artifacts by reference, never inline source code in JSON or YAML records.

There are three user-facing config kinds:

```text
EnvironmentConfig  describes how an environment accepts candidates and produces observations
MethodConfig       describes the user-owned controller/engine optimization method
StudyConfig        binds one method to one environment under an objective and budget
```

The internal expanded file remains:

```text
StudySpec          canonical execution/audit representation generated by OptPilot
```

Most users should write `StudyConfig`, `EnvironmentConfig`, and `MethodConfig`. They should not hand-write `StudySpec` unless they are developing OptPilot itself.

## 2. Conceptual Workflow

Every study follows the same loop:

```text
StudyConfig selects EnvironmentConfig and MethodConfig
OptPilot builds initial study state
Controller reads study state and evidence
Engine receives study state/evidence and proposes candidate artifacts
OptPilot validates candidate artifacts
OptPilot materializes candidates into trial workspaces or runtime payloads
Environment evaluator runs
OptPilot extracts metrics, structured records, saved files, and resource data
OptPilot records official Observations
Controller/engine receives Observations and proposes the next candidates
Study stops when budget or stopping policy is reached
```

The optimization method does not read raw environment-specific output directly by default. It receives OptPilot's normalized evidence:

```json
{
  "trial_id": "trial-123",
  "artifact_id": "artifact-456",
  "status": "success",
  "metric_values": {
    "score": 0.91,
    "runtime": 12.4
  },
  "artifacts": [
    {"path": ".../metrics.json"},
    {"path": ".../stdout.log"}
  ],
  "event_summary": {}
}
```

This is what makes different environments usable with different optimization methods.

## 3. File Resolution Rules

Config files may be YAML or JSON. Examples use YAML.

Relative paths are resolved relative to the config file containing the path, except paths inside generated candidate artifacts, which are relative to the candidate root declared by the environment.

Import references use one of these forms:

```text
module:function
module:Class
python:module:Class
builtin.name
```

The `python:` prefix is required for pluggable OptPilot components such as custom engines, controllers, adapters, metric extractors, or backends. Plain `module:function` is allowed for environment callables.

## 4. Common Top-Level Fields

All config files share these fields:

| Field | Required | Allowed values | Meaning |
| --- | --- | --- | --- |
| `apiVersion` | yes | `optpilot.io/v3alpha1` | Schema version. |
| `kind` | yes | `EnvironmentConfig`, `MethodConfig`, `StudyConfig` | Document type. |
| `id` | required except `StudyConfig` | string | Stable arbitrary label for reuse. |
| `description` | no | string | Human-readable description. |
| `tags` | no | list of strings | Labels for search and organization. |

`id`, `description`, and `tags` are metadata. They do not point to files or Python objects.

## 5. EnvironmentConfig

`EnvironmentConfig` describes the target environment contract:

- how to invoke evaluation
- what shape of candidate it accepts
- how to prepare a workspace if needed
- how to extract official metrics
- what extra files should be saved and what structured records should be extracted

Minimal shape:

```yaml
apiVersion: optpilot.io/v3alpha1
kind: EnvironmentConfig
id: my-environment

evaluate:
  type: command
  command: ["{python}", "evaluate.py", "--metrics", "{metrics_file}"]

candidate:
  type: files

metrics:
  source: file
  path: metrics.json
```

### 5.1 `evaluate`

`evaluate` tells OptPilot how the environment evaluator is invoked.

Allowed `evaluate.type` values:

| Value | Use when |
| --- | --- |
| `command` | The environment is evaluated by running a command in a process. This covers CLI tools, scripts, simulators, benchmark harnesses, and workspace-based projects. |
| `python` | The environment exposes a Python callable that can be invoked in process. |
| `custom` | The environment needs a user-owned OptPilot target adapter. |

`command` fields:

| Field | Required | Meaning |
| --- | --- | --- |
| `type` | yes | Must be `command`. |
| `command` | yes | List of argv strings. Shell strings are intentionally avoided in the user-facing schema. |
| `cwd` | no | Working directory relative to the trial workspace. Default `.`. |
| `env` | no | Environment variables to add or override. Values may use placeholders. |
| `timeoutSeconds` | no | Per-evaluation timeout. If omitted, the study execution timeout or OptPilot default is used. |

Allowed command placeholders:

| Placeholder | Meaning |
| --- | --- |
| `{python}` | Python executable used by OptPilot. |
| `{workspace}` | Trial workspace root. |
| `{candidate_root}` | Directory where file candidates are materialized. |
| `{candidate_file}` | Candidate file path when the candidate has exactly one required file. |
| `{candidate}` | Alias for `{candidate_file}` for compatibility with existing benchmark scripts. |
| `{candidate_json}` | JSON file containing the materialized candidate payload, useful for parameter candidates. |
| `{metrics_file}` | Default metrics file path, usually from `metrics.path`. |
| `{instance_file}` | JSON file containing the current evaluation instance. |
| `{trial_id}` | Current trial id. |
| `{study_id}` | Current study id. |

Example:

```yaml
evaluate:
  type: command
  command:
    - "{python}"
    - "evaluate.py"
    - "--candidate"
    - "{candidate_root}"
    - "--instance"
    - "{instance_file}"
    - "--metrics"
    - "{metrics_file}"
  cwd: "."
  timeoutSeconds: 600
```

`python` fields:

| Field | Required | Meaning |
| --- | --- | --- |
| `type` | yes | Must be `python`. |
| `callable` | yes | Python callable in `module:function` form. |
| `pythonPath` | no | Extra import paths. |
| `timeoutSeconds` | no | Per-evaluation timeout. |

The callable receives:

```python
evaluate(candidate, instance, context) -> dict
```

It may return either normalized OptPilot output:

```python
{
    "metric_values": {"score": 0.91},
    "status": "success",
    "artifacts": []
}
```

or a flat metric dictionary:

```python
{"score": 0.91, "runtime": 12.4}
```

`custom` fields:

| Field | Required | Meaning |
| --- | --- | --- |
| `type` | yes | Must be `custom`. |
| `implementation` | yes | Adapter implementation, `builtin.*` or `python:module:Class`. |
| `config` | no | Opaque config owned by the custom adapter. |

### 5.2 `candidate`

`candidate` tells OptPilot what kind of object the optimization method must produce.

Allowed `candidate.type` values:

| Value | Use when |
| --- | --- |
| `parameters` | Candidate is structured parameter data such as numbers, categories, booleans, or strings. |
| `files` | Candidate is one or more files, including generated code with imports. |
| `opaque` | Candidate is a binary/checkpoint/external object that OptPilot stores and passes by reference. |

`parameters` fields:

| Field | Required | Meaning |
| --- | --- | --- |
| `type` | yes | Must be `parameters`. |
| `schema` | no | Parameter schema used for validation and for methods that need a declared search space. |

Allowed parameter schema field types:

| Type | Required fields | Optional fields |
| --- | --- | --- |
| `float` | none | `min`, `max`, `default` |
| `int` | none | `min`, `max`, `default` |
| `categorical` | `values` | `default` |
| `bool` | none | `default` |
| `string` | none | `default`, `values` |

Example:

```yaml
candidate:
  type: parameters
  schema:
    buffer_size:
      type: int
      min: 1
      max: 64
    dispatch_rule:
      type: categorical
      values: [fifo, shortest_queue, earliest_due_date]
```

`files` fields:

| Field | Required | Meaning |
| --- | --- | --- |
| `type` | yes | Must be `files`. |
| `root` | no | Destination root inside the trial workspace. Default `.`. |
| `required` | no | Files, relative to `root`, that must be present after materialization. |
| `entrypoint` | no | Entrypoint understood by the environment, such as `solver:solve`. |
| `allow` | no | Glob patterns allowed in candidate artifacts, relative to `root`. |
| `deny` | no | Glob patterns rejected in candidate artifacts, relative to `root`. |

Example:

```yaml
candidate:
  type: files
  root: "."
  required: ["solver.py"]
  entrypoint: "solver:solve"
  allow: ["**/*.py", "pyproject.toml"]
  deny: ["data/**", "evaluator.py"]
```

Generated code is stored by reference. A candidate artifact may include many files:

```json
{
  "artifact_id": "artifact-017",
  "artifact_kind": "files",
  "spec": {
    "entrypoint": "solver:solve",
    "files": [
      {
        "path": "solver.py",
        "contentRef": "artifacts/artifact-017/files/solver.py",
        "sha256": "..."
      },
      {
        "path": "helpers.py",
        "contentRef": "artifacts/artifact-017/files/helpers.py",
        "sha256": "..."
      }
    ]
  }
}
```

The source code itself is not embedded in the JSON.

`opaque` fields:

| Field | Required | Meaning |
| --- | --- | --- |
| `type` | yes | Must be `opaque`. |

Opaque candidates are used for checkpoints, learned policies, binary assets, external service handles, or any artifact that a custom adapter/materializer understands.

### 5.3 `workspace`

`workspace` is optional. It is used when evaluation requires files on disk.

This is the trial evaluation workspace, not the controller or engine's own working directory. For each trial, OptPilot creates a clean workspace, copies declared environment files into it, materializes the candidate there, runs the evaluator, and then records metrics/evidence.

Fields:

| Field | Required | Meaning |
| --- | --- | --- |
| `copy` | no | Files or directories copied into each trial workspace before candidate materialization. |
| `readonly` | no | Glob patterns that must not be modified by the candidate or evaluator. OptPilot snapshots them before/after when supported. |

`workspace.copy` entries:

| Field | Required | Meaning |
| --- | --- | --- |
| `from` | yes | Source file or directory path. |
| `to` | yes | Destination path inside the trial workspace. |

Example:

```yaml
workspace:
  copy:
    - from: resource/Frontier-Engineering/tasks/pid
      to: "."
  readonly:
    - "data/**"
    - "evaluate.py"
```

There is no `writable` field. Files are writable unless they are declared read-only, and candidate destinations are writable by definition.

### 5.4 `metrics`

`metrics` tells OptPilot how to extract official scalar metrics from the evaluator output.

Allowed `metrics.source` values:

| Value | Use when |
| --- | --- |
| `return` | A Python evaluator returns metrics directly. |
| `file` | The evaluator writes a JSON metrics file. |
| `stdout` | The evaluator prints JSON metrics to stdout. |
| `sqlite` | Metrics are extracted by SQL query from a SQLite database. |
| `custom` | Metrics need a user-owned extractor. |

Common optional field:

| Field | Required | Meaning |
| --- | --- | --- |
| `keys` | no | Declared metric keys expected from the extractor. Useful for validation and UI. |

`return` fields:

```yaml
metrics:
  source: return
  keys: [score, runtime]
```

Valid only with `evaluate.type: python` or a custom adapter that returns data.

`file` fields:

| Field | Required | Meaning |
| --- | --- | --- |
| `source` | yes | Must be `file`. |
| `path` | yes | JSON file path relative to trial workspace. |
| `keys` | no | Expected metric keys. |

Example:

```yaml
metrics:
  source: file
  path: metrics.json
  keys: [score, runtime]
```

`stdout` fields:

```yaml
metrics:
  source: stdout
  keys: [score]
```

`sqlite` fields:

| Field | Required | Meaning |
| --- | --- | --- |
| `source` | yes | Must be `sqlite`. |
| `database` | yes | SQLite database path relative to trial workspace. |
| `query` | yes | Query returning one row. Numeric columns become metric values. |
| `keys` | no | Expected metric keys. |

Example:

```yaml
metrics:
  source: sqlite
  database: results.db
  query: "select best_score as score, elapsed_seconds as runtime from summary limit 1"
  keys: [score, runtime]
```

`custom` fields:

| Field | Required | Meaning |
| --- | --- | --- |
| `source` | yes | Must be `custom`. |
| `implementation` | yes | Metric extractor, `python:module:Class`. |
| `config` | no | Opaque extractor config. |

OptPilot accepts either normalized metrics:

```json
{"metric_values": {"score": 0.91}}
```

or flat numeric metrics:

```json
{"score": 0.91, "runtime": 12.4}
```

### 5.5 `filesToSave`

`filesToSave` is optional. It declares extra files or directories to preserve as raw evidence.

OptPilot saves these files but does not promise to understand their internal structure. Controllers and engines may see that the files exist, but they should not rely on reading arbitrary raw files directly.

Shape:

```yaml
filesToSave:
  - "metrics.json"
  - "artifacts.json"
  - "logs/**"
```

Each entry is a path or glob relative to the trial workspace. OptPilot infers whether it is a file, directory, log, JSON file, database, or other artifact when possible.

### 5.6 `recordsToExtract`

`recordsToExtract` is optional. It declares structured records that OptPilot should parse and expose through the evidence view.

Metrics answer "how good was this trial?" Extracted records answer "what happened during this trial?"

Allowed `recordsToExtract[].source` values:

| Value | Meaning |
| --- | --- |
| `jsonl` | Read record objects from a JSONL file. |
| `csv` | Read record rows from a CSV file. |
| `sqlite_table` | Read rows from a SQLite table. |
| `sqlite_query` | Read rows from a SQLite query. |
| `custom` | Use a user-owned record extractor. |

Fields:

| Field | Required | Meaning |
| --- | --- | --- |
| `name` | yes | Record stream name used in evidence queries. |
| `source` | yes | One of the allowed record sources. |
| `path` | yes except custom | File or database path relative to workspace. |
| `table` | only for `sqlite_table` | Table name. |
| `query` | only for `sqlite_query` | SQL query. |
| `implementation` | only for `custom` | Record extractor, `python:module:Class`. |
| `config` | no | Opaque extractor config for custom extractors. |

Example:

```yaml
recordsToExtract:
  - name: machine_events
    source: sqlite_table
    path: simulation.db
    table: event_log
```

A file can appear in both `filesToSave` and `recordsToExtract`. For example, saving `simulation.db` preserves the database for audit, while extracting `event_log` makes selected rows queryable by controllers and engines.

## 6. MethodConfig

`MethodConfig` describes the optimization method. OptPilot does not implement Bayesian optimization, RL, metaheuristics, or LLM code evolution as platform requirements. Those live in user-owned engines and controllers.

Minimal shape:

```yaml
apiVersion: optpilot.io/v3alpha1
kind: MethodConfig
id: my-method

engine:
  implementation: python:my_lab.engines:MyEngine
```

Fields:

| Field | Required | Meaning |
| --- | --- | --- |
| `engine` | yes | Component that proposes candidate artifacts and observes trial results. |
| `controller` | no | Component that decides which engine action to take. Defaults to `builtin.single_engine_controller`. |

### 6.1 `engine`

Fields:

| Field | Required | Meaning |
| --- | --- | --- |
| `implementation` | yes | `builtin.reference_random_search` or `python:module:Class`. |
| `config` | no | Opaque method-specific config owned by the engine. |

Engine protocol:

```python
propose(n_candidates, study_state, evidence_view=None) -> list[artifact]
observe(observations) -> None
```

or lifecycle protocol:

```python
start(engine_input) -> handle
poll(handle) -> status
finalize(handle) -> list[artifact] | {"artifacts": [...]}
```

OptPilot infers which protocol the engine implements. No YAML field is needed for `sync` versus `lifecycle`.

The engine receives:

- current best metric
- completed trial count
- candidate schema or candidate file policy
- objective definition
- normalized observations from previous trials
- evidence view for artifacts, extracted records, logs, and prior observations
- runtime context for writing candidate artifact files by reference

The engine returns candidate artifacts. OptPilot, not the engine, owns official final evaluation.

### 6.2 `controller`

Fields:

| Field | Required | Meaning |
| --- | --- | --- |
| `implementation` | no | `builtin.single_engine_controller` or `python:module:Class`. |
| `config` | no | Opaque controller-specific config. |

If omitted:

```yaml
controller:
  implementation: builtin.single_engine_controller
```

Controller protocol:

```python
decide(study_state, engines, evidence_view=None) -> Action
```

The default controller simply asks the configured engine for the next batch until budget is exhausted.

## 7. StudyConfig

`StudyConfig` binds one method to one environment and defines objective, instances, budget, execution, evidence, and reproducibility.

Minimal shape:

```yaml
apiVersion: optpilot.io/v3alpha1
kind: StudyConfig
name: my-study

environment: environments/my-env.yaml
method: methods/my-method.yaml

objective:
  metric: score
  direction: maximize

budget:
  maxTrials: 50
```

### 7.1 References

Fields:

| Field | Required | Meaning |
| --- | --- | --- |
| `name` | yes | Study name. |
| `environment` | yes | Path to an `EnvironmentConfig`, or inline `EnvironmentConfig` object. |
| `method` | yes | Path to a `MethodConfig`, or inline `MethodConfig` object. |

Path references are preferred because environments and methods should be reusable.

### 7.2 `objective`

Fields:

| Field | Required | Allowed values | Meaning |
| --- | --- | --- | --- |
| `metric` | yes | string | Metric key extracted by the environment. |
| `direction` | yes | `maximize`, `minimize` | Optimization direction. |
| `aggregation` | no | `mean`, `median`, `min`, `max`, `sum`, `last` | How to aggregate across instances/seeds. Default `mean`. |

Example:

```yaml
objective:
  metric: score
  direction: maximize
```

`metric` must correspond to a key in normalized `metric_values`. If `metrics.json` contains `{"score": 0.91}`, then `metric: score` is valid.

### 7.3 `instances`

`instances` defines the evaluation cases. If omitted, OptPilot uses one empty instance.

Allowed `instances.source` values:

| Value | Meaning |
| --- | --- |
| `none` | One empty instance. |
| `inline` | One inline instance object. |
| `files` | One or more instance files. |
| `sampler` | User-owned sampler generates instances. |

Shapes:

```yaml
instances:
  source: none
```

```yaml
instances:
  source: inline
  value:
    demand_mean: 120
    seed: 1
```

```yaml
instances:
  source: files
  paths:
    - instances/case_a.yaml
    - instances/case_b.yaml
```

```yaml
instances:
  source: sampler
  implementation: python:my_lab.samplers:DemandSampler
  config:
    demand_mean: [100, 140]
  count: 20
```

### 7.4 `budget`

Fields:

| Field | Required | Meaning |
| --- | --- | --- |
| `maxTrials` | yes | Maximum official OptPilot trial evaluations. |
| `maxWallClockSeconds` | no | Maximum wall-clock runtime for the study. |
| `maxFailures` | no | Maximum failed trials before stopping. |

### 7.5 `execution`

Optional. If omitted, OptPilot uses local execution with parallelism `1`.

Fields:

| Field | Required | Allowed values | Meaning |
| --- | --- | --- | --- |
| `backend` | no | `local`, `local_subprocess`, `custom` | Execution backend. |
| `implementation` | only if backend is `custom` | `python:module:Class` | Custom backend implementation. |
| `parallelism` | no | positive integer | Candidate trials evaluated concurrently. |
| `timeoutSeconds` | no | positive integer | Default timeout for trials. |
| `retry.maxRetries` | no | non-negative integer | Trial retry count. |

Example:

```yaml
execution:
  backend: local_subprocess
  parallelism: 4
  timeoutSeconds: 600
  retry:
    maxRetries: 1
```

### 7.6 `evidence`

Optional. If omitted, OptPilot stores standard evidence under the project run directory.

Fields:

| Field | Required | Allowed values | Meaning |
| --- | --- | --- | --- |
| `outputDir` | no | path | Directory where run evidence is stored. |
| `level` | no | `minimal`, `standard`, `full` | Retention preset. Default `standard`. |

Retention levels:

| Level | Meaning |
| --- | --- |
| `minimal` | Store spec, summary, observations, and final artifacts. |
| `standard` | Store standard run records, metrics, logs, artifacts, lineage, and environment snapshot. |
| `full` | Store all standard evidence plus prompts, intermediate files, controller decisions, engine snapshots, and extracted records. |

### 7.7 `reproducibility`

Optional.

Fields:

| Field | Required | Meaning |
| --- | --- | --- |
| `seed` | no | Global deterministic seed. |

Example:

```yaml
reproducibility:
  seed: 7
```

## 8. Mapping To Internal StudySpec

OptPilot compiles authoring configs to the internal `StudySpec`.

| User-facing field | Internal field |
| --- | --- |
| `EnvironmentConfig.evaluate` | `target.adapter` |
| `EnvironmentConfig.candidate` | `artifacts.primaryArtifact` and materialization/validation defaults |
| `EnvironmentConfig.workspace` | materialization plan config |
| `EnvironmentConfig.metrics` | target adapter metric extraction config |
| `EnvironmentConfig.filesToSave` | target adapter artifact collection config |
| `EnvironmentConfig.recordsToExtract` | structured evidence extraction config |
| `MethodConfig.controller` | `controllers` |
| `MethodConfig.engine` | `engines` |
| `StudyConfig.objective` | `objective.primaryMetric` |
| `StudyConfig.instances` | `evaluationScope` |
| `StudyConfig.budget` | `stopping` |
| `StudyConfig.execution` | `execution` |
| `StudyConfig.evidence` | `evidence` |
| `StudyConfig.reproducibility` | `reproducibility` |

The generated `StudySpec` is recorded in the run evidence for audit and reproducibility.

## 9. Usage Examples

This section shows how representative OptPilot use cases are expressed without adding extra config concepts.

### 9.1 Direct Parameter Search Over A Simulator

Environment:

```yaml
apiVersion: optpilot.io/v3alpha1
kind: EnvironmentConfig
id: factory-simulator

evaluate:
  type: python
  callable: my_factory.sim:evaluate

candidate:
  type: parameters
  schema:
    buffer_size:
      type: int
      min: 1
      max: 64
    dispatch_rule:
      type: categorical
      values: [fifo, shortest_queue, earliest_due_date]

metrics:
  source: return
  keys: [throughput, cycle_time]
```

Method:

```yaml
apiVersion: optpilot.io/v3alpha1
kind: MethodConfig
id: random-search

engine:
  implementation: builtin.reference_random_search
```

Study:

```yaml
apiVersion: optpilot.io/v3alpha1
kind: StudyConfig
name: factory-direct-search

environment: environments/factory-simulator.yaml
method: methods/random-search.yaml

objective:
  metric: throughput
  direction: maximize

instances:
  source: files
  paths: [instances/factory_case.yaml]

budget:
  maxTrials: 100
```

The engine proposes parameter dictionaries. OptPilot passes them to the simulator callable and records official observations.

### 9.2 Bayesian Optimization Over Structured Inputs

The environment is the same as direct parameter search. Only the method changes.

```yaml
apiVersion: optpilot.io/v3alpha1
kind: MethodConfig
id: optuna-tpe

engine:
  implementation: python:my_methods.optuna_engine:OptunaEngine
  config:
    sampler: TPESampler
    batchSize: 4
```

Study:

```yaml
apiVersion: optpilot.io/v3alpha1
kind: StudyConfig
name: factory-bo

environment: environments/factory-simulator.yaml
method: methods/optuna-tpe.yaml

objective:
  metric: throughput
  direction: maximize

budget:
  maxTrials: 80
```

OptPilot does not implement Optuna. The Optuna engine reads prior observations from OptPilot evidence and proposes new parameter candidates.

### 9.3 Metaheuristic Search

```yaml
apiVersion: optpilot.io/v3alpha1
kind: MethodConfig
id: evolutionary-search

engine:
  implementation: python:my_methods.evolution:EvolutionaryEngine
  config:
    populationSize: 32
    mutationRate: 0.15
    batchSize: 8
```

The same `EnvironmentConfig` and `StudyConfig` pattern applies. The algorithm-specific fields remain inside `engine.config`.

### 9.4 Reinforcement Learning With Rollout Workers

For RL, the candidate may be an opaque policy checkpoint, a file bundle containing training scripts, or a file bundle containing a trained policy. The engine owns training and rollout coordination. OptPilot owns official evaluation of returned candidates.

There are two different observation concepts:

```text
RL step observation       what the policy receives from env.step(action)
OptPilot Observation     the official trial result after training/evaluation finishes
```

The RL step observation is defined by the protected RL environment code, not by OptPilot's `Observation` object. If an LLM engine writes training scripts, those scripts should import or wrap the protected environment. OptPilot can copy environment files and observation/action-space documentation into the trial workspace and mark them read-only.

Environment:

```yaml
apiVersion: optpilot.io/v3alpha1
kind: EnvironmentConfig
id: warehouse-rl-training

evaluate:
  type: command
  command:
    - "{python}"
    - "run_training_and_eval.py"
    - "--candidate"
    - "{candidate_root}"
    - "--metrics"
    - "{metrics_file}"
  timeoutSeconds: 3600

candidate:
  type: files
  root: candidate
  required:
    - "train.py"
    - "policy.py"
  allow:
    - "**/*.py"

workspace:
  copy:
    - from: rl_benchmark/
      to: "."
  readonly:
    - "env/**"
    - "run_training_and_eval.py"
    - "observation_space.json"
    - "action_space.json"

metrics:
  source: file
  path: metrics.json
  keys: [mean_reward, success_rate, training_time]

filesToSave:
  - "rollout_logs/**"
  - "trained_policy/**"
```

Method:

```yaml

  implementation: python:my_methods.rllib_engine:RLLibTrainingEngine
  config:
    trainIterations: 100

```


The trial workspace for one candidate may look like this:

```text
trial-001/
  env/                         protected RL environment
  run_training_and_eval.py      protected evaluator/training harness
  observation_space.json        protected description of step observations
  action_space.json             protected description of actions
  candidate/                    LLM- or engine-generated candidate files
    train.py
    policy.py
  metrics.json                  official trial metrics written by evaluator
```

The candidate training code receives per-step RL observations through the environment API. OptPilot receives only the final trial metrics and records the official study-level observation.

### 9.7 Nested Optimization

Nested optimization means the artifact being evaluated is itself a solver, trainer, or workflow. OptPilot only needs to know how to evaluate that artifact.

Environment:

```yaml
apiVersion: optpilot.io/v3alpha1
kind: EnvironmentConfig
id: solver-workflow-eval

evaluate:
  type: command
  command:
    - "{python}"
    - "run_outer_evaluation.py"
    - "--workflow"
    - "{candidate_root}"
    - "--metrics"
    - "{metrics_file}"
  timeoutSeconds: 3600

candidate:
  type: files
  root: workflow
  required: ["workflow/main.py"]
  entrypoint: "main:run"
  allow: ["workflow/**"]

metrics:
  source: file
  path: metrics.json
  keys: [outer_score, cost]

filesToSave:
  - "inner_runs/**"
  - "metrics.json"
```

Study:

```yaml
apiVersion: optpilot.io/v3alpha1
```

The candidate is a workflow implementation. The environment command may run an inner solver or trainer internally, but OptPilot records only the official outer evaluation as the study observation.

### 9.8 Existing Workspace Benchmark Projects

Projects like the resource examples often have this shape:

```text
existing project files
editable candidate files
frozen evaluator script
data/database files
metrics.json or results database
```

They fit the same `command + files + file/sqlite metrics` pattern.

```yaml
apiVersion: optpilot.io/v3alpha1
kind: EnvironmentConfig
id: existing-workspace-task

evaluate:
  type: command
  command:
    - "{python}"
    - "evaluation/run.py"
    - "--candidate-dir"
    - "{candidate_root}"
    - "--metrics"
    - "{metrics_file}"
  cwd: "."

candidate:
  type: files
  root: "."
  required: ["agent.py"]
  allow: ["**/*.py"]
  deny: ["evaluation/**", "data/**"]

workspace:
  copy:
    - from: resource/some-existing-project/task_a
      to: "."
  readonly:
    - "evaluation/**"
    - "data/**"

metrics:
  source: file
  path: metrics.json
  keys: [score]

recordsToExtract:
  - name: simulation_events
    source: sqlite_table
    path: results/simulation.db
    table: event_log

filesToSave:
  - "results/**"
  - "metrics.json"
```

This avoids writing a project-specific OptPilot adapter for every normal workspace benchmark.

## 10. Field Necessity Summary

The final user-facing schema keeps only three real choices.

Environment choices:

```text
evaluate.type: command | python | custom
candidate.type: parameters | files | opaque
metrics.source: return | file | stdout | sqlite | custom
optional workspace.copy / workspace.readonly
optional filesToSave
optional recordsToExtract
```

Method choices:

```text
engine.implementation
optional controller.implementation
```

Study choices:

```text
environment reference
method reference
objective metric and direction
instances
budget
execution/evidence/reproducibility overrides
```

Everything else is inferred or belongs inside explicit user-owned `config` fields.

This gives OptPilot enough information to connect optimization methods to environments without forcing every user to understand internal adapters, materializers, validators, schedulers, or evidence-store details.
