# OptPilot Self-Contained Design

Status note: this document is the high-level design narrative. The current
user-facing configuration contract is defined in
`docs/config_files_v3alpha.md` using `StudyConfig`, `EnvironmentConfig`, and
`MethodConfig`. `StudySpec` is now the internal expanded execution/audit
representation generated by OptPilot, not the normal file users should write.

## 1. Executive Summary

OptPilot is a platform for running optimization studies over target environments.

It is not intended to be a Bayesian optimization library, an RL framework, an LLM agent framework, a simulator, a scheduler, or a sandbox runtime. Instead, OptPilot provides the stable layer that lets those systems work together under one reproducible study protocol.

The central idea is:

```text
User-owned controllers and engines propose or update artifacts.
OptPilot materializes and evaluates those artifacts against protected targets.
OptPilot records normalized observations, artifacts, lineage, policy, and provenance.
Controllers and engines use stored evidence to decide what to do next.
```

This lets the same platform support:

- direct parameter search over a simulator
- Bayesian optimization over structured inputs
- meta-heuristic search
- reinforcement learning with rollout workers
- LLM-guided code evolution
- workspace-based code optimization against frozen verifiers
- benchmark suites whose tasks declare editable files, read-only files, runtime commands, and metric files
- LLM-supervised RL or Bayesian optimization
- nested optimization, where the artifact under study is itself a solver, trainer, or workflow

The most important design boundary is:

```text
OptPilot owns protocol, execution handoff, evidence, policy, and reproducibility.
Users and external packages own the actual optimization algorithms and domain systems.
```

## 2. Why OptPilot Exists

Optimization projects often become collections of custom scripts:

- one script for running a simulator
- one script for sampling parameters
- one script for launching training
- one script for collecting metrics
- one folder of logs
- one notebook for analysis
- one ad hoc format for reproducing results

This works for one experiment, but it becomes fragile when users want to compare methods, reuse evidence, run in parallel, audit outputs, or combine LLMs, RL, and classical optimizers.

OptPilot solves the shared platform problem:

- define the study
- define the target
- define what artifact is being improved
- run bounded trials
- normalize observations
- store all evidence
- preserve lineage
- expose evidence back to controllers and engines
- keep implementation details pluggable

The platform should make experiments scientifically defensible and operationally repeatable without forcing all users into one algorithm stack.

## 3. System Boundary

### 3.1 OptPilot Owns

OptPilot should implement and maintain:

- user-facing config schema, loading, validation, defaults, and compilation to internal `StudySpec`
- study orchestration
- target adapter interface
- controller interface
- engine interface
- execution backend interface
- artifact model
- materialization interface
- validation interface
- trial model
- observation model
- evidence store interface
- local reference evidence store
- resource profile model
- sandbox spec model
- access and mutation policy model
- failure and status normalization
- lineage and provenance records
- reproducibility records
- resume and branch metadata

These are platform contracts. They define what OptPilot means by a study, a trial, an artifact, an observation, and a reproducible run.

### 3.2 Users and External Tools Own

Users or existing packages should provide:

- Bayesian optimization algorithms
- meta-heuristic algorithms
- RL training loops
- LLM agent loops
- model provider integrations
- domain simulators
- service endpoints
- cluster schedulers
- container runtimes
- strong sandboxing systems
- GPU placement systems

OptPilot should integrate with these systems, not replace them.

### 3.3 Lessons From Existing Resource Projects

Two existing projects in `resource/` clarify the integration target OptPilot should support:

- `resource/OptPilot-main`
- `resource/Frontier-Engineering`

Both show that real environments are often not simple functions. They are usually workspace-based systems:

- a working directory is prepared
- one or more editable source files are placed into it
- fixed context files, databases, references, or benchmark assets are copied in
- a frozen simulator or verifier is invoked
- metrics are returned through a Python object, `metrics.json`, stdout, or a database
- extra artifacts such as logs, generated reports, SQLite databases, checkpoints, and result files must be preserved

This implies a key platform requirement:

```text
OptPilot should make workspace-based target integration first-class.
```

Users should not have to hand-write a new adapter for every existing environment that follows the common pattern:

```text
candidate artifact + prepared workspace + frozen evaluator command -> metrics + artifacts
```

The platform should therefore provide reusable support for:

- code and file-bundle artifacts
- workspace materialization
- declarative evaluator commands
- declarative metric extraction
- read-only file checks
- SQL/database artifact discovery
- environment metadata importers
- runtime environment declarations

The project should still not own the optimization algorithm. For example, OpenEvolve, Optuna, an LLM coding agent, or a custom planner remains user-owned. OptPilot owns the study protocol around those tools.

### 3.4 Reference Implementations

OptPilot may include small reference components for testing and demonstration:

- local backend
- local evidence store
- Python callable target adapter
- CLI target adapter
- parameter passthrough materializer
- simple schema or bounds validator
- reference random-search engine

These should be clearly labeled as reference components. They are not the reason the platform exists.

## 4. Core Loop

The core loop is:

```text
Study starts
Controller reads study state and evidence
Controller chooses an action
Engine proposes or updates an artifact
OptPilot validates the artifact
OptPilot materializes the artifact into a runtime payload
Execution backend runs a trial against the target adapter
Target adapter returns raw results and artifacts
OptPilot normalizes results into observations
Evidence store records trials, observations, artifacts, lineage, and provenance
Controller or engine uses evidence for the next decision
Study stops when budget or policy says so
```

This loop is intentionally general. It can represent a short parameter evaluation, a long RL training job, an LLM code-evolution attempt, or a nested solver evaluation.

## 5. Main Objects

### 5.1 Study

A `Study` is the top-level optimization effort.

It defines:

- target environment
- objective
- evaluation scope
- optimizable artifacts
- controllers
- engines
- execution policy
- resource policy
- sandbox policy
- evidence policy
- reproducibility policy
- stopping rules

A study is broader than an optimizer. It is the complete experimental protocol.

### 5.2 Target Environment

A `TargetEnvironment` is the protected system against which performance is measured.

Examples:

- factory simulator
- warehouse routing simulator
- scheduling benchmark
- workspace benchmark
- frozen verifier over editable code
- HTTP evaluation service
- Gymnasium environment
- OpenEnv environment
- compound evaluation harness

The target environment is not the thing being optimized. It should generally be immutable during a study.

Target fields:

```yaml
target:
  targetId: factory-sim-v1
  adapter:
    type: python
    implementation: builtin.python_callable
    config:
      module: my_factory.sim
      callable: evaluate
  accessPolicy: SchemaAware
  mutationPolicy: NoMutation
  runtimeContract:
    timeoutSeconds: 600
```

For workspace-based targets, the `EnvironmentConfig` describes the command,
candidate files, saved files, and metric source:

```yaml
apiVersion: optpilot.io/v3alpha1
kind: EnvironmentConfig
id: frontier-pid-tuning

evaluate:
  type: command
  command: ["{python}", "frontier_eval/run_eval.py", "--candidate", "{candidate}", "--metrics-out", "{metrics_file}"]
  timeoutSeconds: 300

candidate:
  type: files
  required: [scripts/init.py]

workspace:
  readonly: ["frontier_eval/**"]

metrics:
  source: file
  path: metrics.json

filesToSave:
  - artifacts.json
```

The workspace benchmark adapter should not require users to predict every metric or artifact field in advance. It should provide a stable envelope, discover task metadata, run the evaluator, and preserve raw files.

### 5.3 Evaluation Scope

The evaluation scope defines what population of cases the study optimizes against.

Supported conceptual modes:

- `FixedInstance`: optimize one case
- `InstanceSet`: optimize over a fixed benchmark set
- `Distribution`: optimize over sampled instances
- `Curriculum`: optimize over staged distributions or difficulty levels

Example:

```yaml
evaluationScope:
  mode: InstanceSet
  definition:
    instanceRefs:
      - instances/factory_a.yaml
      - instances/factory_b.yaml
      - instances/factory_c.yaml
```

### 5.4 Objective

The objective defines how trial performance is judged.

It should support:

- primary metric
- maximize or minimize direction
- constraints
- secondary metrics
- aggregation over instances, seeds, or episodes
- cost-aware scoring

Example:

```yaml
objective:
  primaryMetric:
    name: throughput
    direction: maximize
  constraints:
    - metric: lateness
      operator: <=
      value: 0.05
  secondaryMetrics:
    - cycle_time
    - energy_cost
  aggregation:
    mode: mean
```

### 5.5 Optimizable Artifact

An `OptimizableArtifact` is the versioned object being improved.

Examples:

- parameter configuration
- code patch
- single editable code file
- code bundle
- workspace bundle
- solver implementation
- policy checkpoint
- RL training specification
- reward function
- workflow graph

Stable interface:

```python
class OptimizableArtifact:
    artifact_id: str
    artifact_kind: str
    spec: dict
    lineage: dict
    generator_record: dict
    validation_rules: dict
    materialization_plan: dict
```

Example artifact:

```json
{
  "artifact_id": "artifact-001",
  "artifact_kind": "parameter_spec",
  "spec": {
    "buffer_size": 12,
    "dispatch_rule": "shortest_queue"
  },
  "lineage": {
    "parents": []
  },
  "generator_record": {
    "engine_id": "user_bo",
    "strategy": "optuna_tpe"
  },
  "validation_rules": {
    "implementation": "builtin.schema_validation"
  },
  "materialization_plan": {
    "implementation": "builtin.parameter_to_config"
  }
}
```

Workspace-oriented artifacts should be represented explicitly rather than squeezed into a parameter schema.

Single-file code artifact:

```json
{
  "artifact_id": "artifact-code-001",
  "artifact_kind": "code_file",
  "spec": {
    "path": "scripts/init.py",
    "contentRef": "artifacts/artifact-code-001/files/scripts/init.py",
    "sha256": "..."
  },
  "lineage": {
    "parents": ["artifact-code-000"]
  },
  "generator_record": {
    "engine_id": "user_code_engine",
    "strategy": "llm_code_mutation"
  }
}
```

Multi-file bundle artifact:

```json
{
  "artifact_id": "artifact-bundle-001",
  "artifact_kind": "code_bundle",
  "spec": {
    "bundleRef": "artifacts/artifact-bundle-001/files",
    "files": [
      {
        "path": "scheduler.py",
        "contentRef": "artifacts/artifact-bundle-001/files/scheduler.py",
        "sha256": "..."
      },
      {
        "path": "param_estimator.py",
        "contentRef": "artifacts/artifact-bundle-001/files/param_estimator.py",
        "sha256": "..."
      }
    ]
  },
  "lineage": {
    "parents": ["artifact-bundle-000"]
  },
  "generator_record": {
    "engine_id": "llm_factory_optimizer",
    "strategy": "parallel_plan_execution"
  }
}
```

Code artifacts should not inline source code in the artifact JSON. The artifact record should contain identity, lineage, entrypoints, file paths, file hashes, and references to files stored in the evidence or artifact store. This keeps records compact, supports generated multi-file modules and imports, and avoids duplicating large code blobs in JSONL evidence.

The alpha provides `optpilot.code_artifacts.CodeArtifactStore` for this common
engine-side workflow. During a run, user-owned engines receive
`study_state["runtime_context"]["artifact_store_dir"]`, so generated files can
be stored in the run's evidence directory:

```python
from optpilot.code_artifacts import CodeArtifactStore

store = CodeArtifactStore(
    study_state["runtime_context"]["artifact_store_dir"],
    content_ref_mode=study_state["runtime_context"]["artifact_content_ref_mode"],
)
artifact = store.store_directory(
    "engine-output/candidate-17",
    artifact_id="artifact-code-017",
    entrypoint="solver:solve",
    generator_record={
        "engine_id": "llm_supervisor",
        "strategy": "code_evolution",
        "prompt_record_id": "prompt-123",
    },
)
```

This produces a manifest-only `code_bundle` with `contentRef`, `sha256`, and
`sizeBytes` for each file. It also supports `content_ref_mode="relative"` for
study-local artifact stores. The helper deliberately stores files and produces
metadata; it does not run optimization or evaluate candidates.

For LLM-based engines, the alpha also provides prompt/model provenance helpers:

```python
from optpilot.provenance import PromptStore, build_generator_record, build_model_record

prompt_store = PromptStore(study_state["runtime_context"]["prompt_store_dir"])
prompt_record = prompt_store.store_prompt(
    messages=[
        {"role": "system", "content": "You are improving solver.py."},
        {"role": "user", "content": "Return a valid code bundle."},
    ],
)
model_record = build_model_record(
    provider="example",
    model="code-model-v1",
    parameters={"temperature": 0.2},
)
generator_record = build_generator_record(
    engine_id="llm_engine",
    strategy="code_evolution",
    prompt_record=prompt_record,
    model_record=model_record,
)
```

The artifact can then carry `generator_record` while prompt text lives in the
prompt store with `contentRef` and `sha256`. OptPilot records provenance; users
still own prompt construction, model calls, and code generation.

The referenced files should contain only study-owned candidate content. Protected benchmark assets, frozen verifiers, seed databases, and reference files belong to the target or materialization context, not the mutable artifact.

### 5.6 Materialization

Materialization converts an artifact into what the target can run.

Examples:

- parameters become JSON config
- parameters become CLI flags
- generated code is copied into a workspace
- a code bundle is written into an evaluator workspace
- baseline files and seed databases are copied into a trial workspace
- a candidate file is placed at the evaluator's expected destination path
- a checkpoint is restored
- a training spec becomes a job directory

OptPilot owns the materialization protocol because this is the boundary between optimization representation and execution representation.

Workspace materialization should support:

- creating a clean trial workspace
- copying declared input files and directories
- writing candidate files at declared paths
- setting executable permissions when needed
- recording hashes of candidate files
- recording hashes of read-only files before evaluation
- preserving a materialization manifest

For Frontier-style tasks, materialization should read task-local metadata such as:

```text
frontier_eval/initial_program.txt
frontier_eval/candidate_destination.txt
frontier_eval/copy_files.txt
frontier_eval/readonly_files.txt
frontier_eval/agent_files.txt
frontier_eval/artifact_files.txt
frontier_eval/constraints.txt
```

For `resource/OptPilot-main` style tasks, materialization should support a simpler project-runner pattern:

```text
working_dir/
  scheduler.py or strategy.py
  param_estimator.py
  database.db
```

These two cases should share the same platform abstraction: a `workspace_bundle` target context plus a mutable `code_file` or `code_bundle` artifact.

### 5.7 Controller

A `Controller` decides what to do next.

Examples:

- fixed workflow controller
- LLM supervisor
- human-in-the-loop controller
- multi-stage controller
- controller that switches from BO to RL

Conceptual interface:

```python
class Controller:
    def decide(self, study_state, evidence_view) -> list[Action]:
        ...
```

OptPilot defines how controllers plug in and what evidence they may inspect. Users provide actual controller logic.

### 5.8 Engine

An `Engine` is a user-owned search, training, or transformation component.

Examples:

- Optuna engine
- BoTorch engine
- DEAP evolutionary engine
- RLlib trainer
- Stable-Baselines trainer
- LLM code mutation engine
- validation engine

General lifecycle interface:

```python
class Engine:
    def start(self, engine_input) -> str:
        ...

    def poll(self, handle) -> dict:
        ...

    def intervene(self, handle, action) -> None:
        ...

    def finalize(self, handle) -> list[dict] | dict:
        ...
```

Lifecycle engines are for long-running or externally orchestrated search processes. They may run Bayesian optimization, evolutionary search, RL training, LLM code mutation, human review, or nested workflows internally. Their authoritative output to OptPilot should still be candidate artifacts.

Default contract:

```text
Engines search.
OptPilot evaluates.
```

In the candidate-producing lifecycle mode, `finalize` returns either a list of candidate artifacts or a dictionary containing an `artifacts` list. OptPilot then validates, materializes, evaluates, and records the official `Observation`.

Engines may also report internal metrics, histories, prompts, logs, checkpoints, and intermediate scores. Those records are useful evidence, but they are not the official study result unless they pass back through OptPilot's evaluator. This keeps comparisons clean across Optuna, OpenEvolve, LLM agents, RL trainers, and custom search code.

Simple synchronous engine interface, useful for lightweight cases:

```python
class CandidateProposalEngine:
    def propose(self, n_candidates, study_state) -> list[dict]:
        ...

    def observe(self, observations) -> None:
        ...
```

OptPilot should support both, but the full platform should grow around lifecycle engines because RL training, BO batches, and nested workflows can be long-running.

### 5.9 Trial

A `Trial` is one bounded execution unit.

Examples:

- evaluate one parameter artifact on one instance
- evaluate one artifact on ten seeds
- train one RL policy for a budget
- run one solver implementation on a benchmark set
- run a nested optimization process and evaluate its final output

Trial shapes:

- `AtomicTrial`
- `BatchTrial`
- `CompoundTrial`

### 5.10 Observation

An `Observation` is the normalized result of a trial.

It includes:

- `trial_id`
- `study_id`
- `artifact_id`
- `target_id`
- `instance_descriptor`
- `status`
- `metric_values`
- `constraint_results`
- `resource_usage`
- `artifacts`
- `event_summary`
- `provenance`

Failures are also observations.

Statuses should include:

- `success`
- `partial`
- `invalid`
- `failed`
- `timeout`
- `cancelled`

### 5.11 Evidence Store

The evidence store records:

- study spec
- run policy
- controller decisions
- engine events
- artifact records
- materialization records
- validation reports
- trial records
- observations
- logs
- traces
- SQL database artifacts
- discovered database schemas
- metric extraction records
- evaluator stdout and stderr
- read-only file violation reports
- generated files
- prompts and model calls when applicable
- lineage and provenance

The evidence store should expose query APIs so controllers and engines can retrieve prior evidence.

Evidence APIs should avoid one bespoke method per possible environment event. They should provide both typed convenience views and a general event query surface.

Conceptual API:

```python
class EvidenceView:
    def summary(self) -> EvidenceSummary: ...
    def observations(self, limit=None, status=None) -> list[dict]: ...
    def artifacts(self, limit=None, artifact_kind=None) -> list[dict]: ...
    def events(
        self,
        event_type=None,
        source_type=None,
        source_id=None,
        trial_id=None,
        artifact_id=None,
        limit=None,
    ) -> list[dict]: ...
```

For SQL-backed evidence, controllers should still use `EvidenceView`. SQL is an evidence-store implementation detail. Common columns such as `trial_id`, `artifact_id`, `status`, `primary_metric_value`, and `created_at` can be indexed, while environment-specific payloads should remain JSON.

Suggested SQL shape:

```text
observations(
  observation_id,
  trial_id,
  artifact_id,
  status,
  primary_metric_name,
  primary_metric_value,
  metric_values_json,
  event_summary_json,
  provenance_json,
  created_at
)

events(
  event_id,
  event_type,
  source_type,
  source_id,
  trial_id,
  artifact_id,
  payload_json,
  created_at
)

artifacts(
  artifact_record_id,
  artifact_id,
  trial_id,
  artifact_kind,
  path,
  metadata_json,
  created_at
)
```

This gives generic controllers enough structure while allowing domain-specific controllers to inspect raw metrics, database schemas, logs, or benchmark artifacts.

## 6. Config Shape

The public config should be declarative and small. Users normally write three
files or inline equivalents:

```text
EnvironmentConfig  describes the candidate type, evaluator, metrics, saved files, and extracted records
MethodConfig       describes the user-owned controller/engine method
StudyConfig        binds one method to one environment under an objective and budget
```

OptPilot compiles these files into the internal `StudySpec` used by the runner.
The complete field list, enum values, and examples live in
`docs/config_files_v3alpha.md`.

Configs should not encode framework-specific details into platform-owned fields.
Instead, user implementations should be referenced through component hooks:

```yaml
implementation: python:my_lab.engines:OptunaEngine
config:
  sampler: tpe
  batchSize: 4
```

## 7. Component Loading

OptPilot should support at least two implementation references:

```yaml
implementation: builtin.python_callable
```

for small reference components, and:

```yaml
implementation: python:my_package.module:ClassName
```

for user-owned components.

This lets OptPilot stay stable while users bring their own frameworks.

## 8. Example 1: Direct Parameter Search Over A Simulator

### Goal

Tune a few simulator parameters for one fixed factory instance.

### OptPilot-Owned Pieces

- study orchestration
- target adapter
- artifact validation
- materialization
- trial execution
- observation and evidence storage

### User-Owned Pieces

- the simulator
- optionally the search engine, if not using a reference fixture

### StudySpec Sketch

```yaml
metadata:
  name: factory-parameter-search

target:
  targetId: factory-sim-v1
  adapter:
    type: python
    implementation: builtin.python_callable
    config:
      module: factory_sim.api
      callable: evaluate
  accessPolicy: SchemaAware
  mutationPolicy: NoMutation

objective:
  primaryMetric:
    name: throughput
    direction: maximize
  aggregation:
    mode: mean

evaluationScope:
  mode: FixedInstance
  definition:
    instanceRef: instances/factory_case_01.yaml

artifacts:
  primaryArtifact:
    kind: parameter_spec
    materializationPlan:
      implementation: builtin.parameter_to_config
    validationRules:
      implementation: builtin.schema_validation
      config:
        enforceBounds: true

controllers:
  - id: controller_main
    implementation: builtin.single_engine_controller

engines:
  - id: user_search
    type: user_owned_search
    implementation: python:my_lab.engines:MySearchEngine
    config:
      batchSize: 8
      searchSpace:
        buffer_size: {type: int, min: 1, max: 20}
        dispatch_rule: {type: categorical, values: [fifo, shortest_queue]}
```

### Why The Design Supports This

The engine proposes `parameter_spec` artifacts. OptPilot validates and materializes those artifacts, evaluates them with the Python target adapter, and records metrics and files as observations.

## 9. Example 2: Bayesian Optimization With Optuna

### Goal

Use Optuna to choose candidate parameters.

### OptPilot-Owned Pieces

- engine interface
- study state and evidence handoff
- trial execution
- evidence recording

### User-Owned Pieces

- Optuna study object
- sampler choice
- search-space interpretation
- optimization algorithm state

### User Engine Sketch

```python
class OptunaEngine:
    def __init__(self, definition, study_spec, rng):
        self.definition = definition
        self.study_spec = study_spec
        self.optuna_study = create_or_load_optuna_study(definition["config"])

    def propose(self, n_candidates, study_state):
        artifacts = []
        for _ in range(n_candidates):
            trial = self.optuna_study.ask()
            spec = {
                "buffer_size": trial.suggest_int("buffer_size", 1, 20),
                "dispatch_rule": trial.suggest_categorical("dispatch_rule", ["fifo", "shortest_queue"])
            }
            artifacts.append({
                "artifact_id": f"artifact-{trial.number}",
                "artifact_kind": "parameter_spec",
                "spec": spec,
                "lineage": {"parents": []},
                "generator_record": {"engine_id": self.definition["id"], "tool": "optuna"}
            })
        return artifacts

    def observe(self, observations):
        for observation in observations:
            if observation["status"] == "success":
                value = observation["metric_values"]["throughput"]
                self.optuna_study.tell(..., value)
```

### StudySpec Sketch

```yaml
engines:
  - id: optuna_main
    type: bayesopt
    implementation: python:my_lab.engines:OptunaEngine
    config:
      sampler: tpe
      batchSize: 4
```

### Why The Design Supports This

OptPilot does not need to know how Optuna works. It only needs artifacts from `propose` and observations passed back to `observe`. Optuna remains user-owned.

## 10. Example 3: Meta-Heuristic Search

### Goal

Use a package such as DEAP, Nevergrad, pymoo, or scipy optimize to propose candidates.

### User-Owned Pieces

- evolutionary population
- mutation and crossover
- selection rule
- archive or search state

### Engine Sketch

```yaml
engines:
  - id: deap_main
    type: metaheuristic
    implementation: python:my_lab.engines:DeapEngine
    config:
      populationSize: 64
      mutationRate: 0.2
      batchSize: 16
```

### Why The Design Supports This

The engine can maintain any internal population representation. OptPilot only requires it to produce artifacts and consume observations. Artifact lineage records parent candidates, which is useful for evolutionary search.

## 11. Example 4: CLI-Based Simulator

### Goal

Evaluate candidates using a simulator that is only available as a command line program.

### OptPilot-Owned Pieces

- CLI adapter
- JSON or file handoff
- stdout and stderr evidence capture
- timeout normalization

### StudySpec Sketch

```yaml
target:
  targetId: factory-cli-v1
  adapter:
    type: cli
    implementation: builtin.cli_target
    config:
      command:
        - python3
        - -m
        - factory_cli.evaluate
        - --artifact
        - "{artifact_path}"
        - --instance
        - "{instance_path}"
        - --output
        - "{output_path}"
      timeoutSeconds: 300
  accessPolicy: InvocationOnly
  mutationPolicy: NoMutation
```

### Why The Design Supports This

OptPilot writes artifact and instance payloads to the trial workspace, invokes the CLI command, reads the result file, and records stdout, stderr, input files, output files, and status.

## 12. Example 5: LLM-Guided Code Evolution

### Goal

Use an LLM to generate or mutate solver code. The solver is the artifact. The target environment remains protected.

### OptPilot-Owned Pieces

- artifact model for code artifacts
- materialization into a controlled workspace
- validation hooks
- sandbox policy declaration
- evidence storage
- prompt/model provenance records

### User-Owned Pieces

- LLM controller or engine
- prompt construction
- model provider
- code mutation strategy
- selection strategy

### Artifact Example

```json
{
  "artifact_kind": "code_module",
  "spec": {
    "bundleRef": "artifacts/artifact-llm-001/files",
    "files": [
      {
        "path": "solver.py",
        "contentRef": "artifacts/artifact-llm-001/files/solver.py",
        "sha256": "..."
      }
    ],
    "entrypoint": "solver:solve"
  },
  "lineage": {
    "parents": ["artifact-previous"]
  },
```

```yaml
apiVersion: optpilot.io/v3alpha1
```

```yaml
apiVersion: optpilot.io/v3alpha1

    path: database.db
```

### SQL Database Evidence

The copied `database.db` should be treated as an artifact. OptPilot should also optionally introspect it and record:

- table names
- table schemas
- row counts
- configured summary queries

The controller or engine can then inspect database evidence through `EvidenceView` without OptPilot knowing all domain-specific tables in advance.

Example configured summary:

```yaml
evidence:
  databaseIntrospection:
    enabled: true
    artifacts:
      - database.db
    summaryQueries:
      factory_order_count: 'SELECT COUNT(*) FROM "order"'

```

### Why The Design Supports This

These apps become ordinary workspace trials. A user-owned LLM or search engine
proposes a `files` artifact by reference; OptPilot copies the baseline files
and database, writes the candidate files, calls the existing runner, extracts
the returned KPI dict, saves configured files, and extracts configured database
records into evidence.

## 16. Example 9: Frontier-Engineering Benchmarks

### Goal

Support Frontier-Eng tasks with minimal manual effort by consuming their existing `frontier_eval/` metadata.

Frontier tasks already declare:

- `initial_program.txt`
- `candidate_destination.txt`
- `copy_files.txt`
- `readonly_files.txt`
- `agent_files.txt`
- `artifact_files.txt`
- `eval_command.txt`
- `eval_cwd.txt`
- `constraints.txt`

### Recommended OptPilot Integration

Use a Frontier metadata importer plus the generic configured-environment path.
The importer emits normal v3alpha configs instead of introducing a separate
Frontier-only public schema:

```yaml
apiVersion: optpilot.io/v3alpha1
kind: EnvironmentConfig
id: frontier-robotics-pid

evaluate:
  type: command
  command:
    - "{python}"
    - "frontier_eval/run_eval.py"
    - "--candidate"
    - "{candidate}"
    - "--metrics-out"
    - "{metrics_file}"
  cwd: "."

candidate:
  type: files
  root: "."
  required:
    - scripts/init.py

workspace:
  readonly:
    - "frontier_eval/**"

metrics:
  source: file
  path: metrics.json
  keys: [combined_score]

filesToSave:
  - artifacts.json
```

The importer should translate Frontier metadata into an OptPilot target contract:

| Frontier Metadata | OptPilot Meaning |
|---|---|
| `initial_program.txt` | initial artifact source |
| `candidate_destination.txt` | materialization destination |
| `copy_files.txt` | workspace input copy plan |
| `readonly_files.txt` | validation/read-only policy |
| `agent_files.txt` | controller/engine context evidence |
| `artifact_files.txt` | `filesToSave` |
| `eval_command.txt` | evaluator command |
| `eval_cwd.txt` | evaluator working directory |
| `metrics.json` | metric extraction source |
| `artifacts.json` | structured artifact extraction source |

### Runtime Support

Frontier tasks may require:

- process execution
- Docker execution
- task-specific Python virtual environments
- GPU access
- host tools such as Octave
- extra datasets or checkpoints

OptPilot should record these requirements in `ResourceProfile` and `SandboxSpec`, then delegate enforcement to execution backends.

Example:

```yaml
execution:
  backend:
    implementation: builtin.local_backend
  defaults:
    resourceProfile:
      cpu: 4
      memoryGiB: 16
      gpu: 0
      timeoutSeconds: 300
    sandboxSpec:
      runtimeType: process
      networkPolicy: disabled
```

For tasks that declare Docker isolation, OptPilot should only expose a Docker backend when the backend actually enforces the declared sandbox contract.

### Relationship To Frontier Algorithms

Frontier-Eng includes algorithm adapters such as OpenEvolve. OptPilot should not copy these algorithms into core. Instead:

- a Frontier/OpenEvolve wrapper can be a user-owned lifecycle engine
- the wrapper should return candidate code artifacts by reference
- OptPilot should perform the official evaluation through the configured target adapter
- OptPilot can record engine snapshots, program history, metrics, prompts, and artifacts

This preserves the project boundary: Frontier/OpenEvolve owns the search; OptPilot owns the study protocol and evidence layer.

### Current Alpha Shape

The alpha provides `optpilot.importers.build_frontier_unified_study_config(...)`.
It reads a benchmark's `frontier_eval/` metadata and emits a normal v3alpha
`StudyConfig` draft using:

- `builtin.configured_environment` internally for evaluator execution
- `builtin.workspace_bundle` for copying benchmark files into a clean trial workspace
- `builtin.workspace_policy` for manifest-only file artifact validation
- a user-owned `python:...` engine placeholder for code generation or search

Example:

```python
from pathlib import Path
import yaml

from optpilot.importers import build_frontier_unified_study_config

benchmark = Path("resource/Frontier-Engineering/benchmarks/Robotics/PIDTuning")
config = build_frontier_unified_study_config(
    benchmark,
    engine_implementation="python:my_lab.engines:FrontierCodeEngine",
    max_trials=20,
)

Path("frontier_pid_study.yaml").write_text(
    yaml.safe_dump(config, sort_keys=False),
    encoding="utf-8",
)
```

The generated engine is intentionally user-owned. It should return `files`
artifact manifests by reference and hash. OptPilot then validates the manifest,
copies Frontier's declared `copy_files`, writes candidate files at
`candidate_destination.txt`, checks `readonly_files.txt`, runs `eval_command.txt`,
and records the official observation from `metrics.json`.

The helper `build_frontier_initial_artifact(...)` can create a manifest for the
task's baseline program. This is useful for smoke tests and for bootstrapping a
user-owned code evolution engine.

The same importer is exposed through the CLI:

```bash
PYTHONPATH=src python3 -m optpilot import-frontier \
  resource/Frontier-Engineering/benchmarks/Robotics/PIDTuning \
  --output frontier_pid_study.yaml
```

## 20. Failure Handling

OptPilot should treat failures as evidence.

Examples:

- invalid artifact
- validation exception
- materialization error
- target exception
- CLI nonzero exit
- evaluator command nonzero exit
- timeout
- partial instance failure
- backend cancellation
- read-only file mutation
- missing metric file
- invalid metric JSON
- missing runtime dependency
- SQL/database introspection failure

A failed trial should still record:

- trial id
- artifact id
- failure phase
- error type
- error message
- traceback or structured detail
- resource and sandbox context
- available logs and files
- evaluator command and working directory
- copied workspace manifest
- read-only violation details
- extracted metric payload, if any
- database schema summary, if available

This is important because user-owned engines, LLM-generated code, and external systems will fail often.

Controllers and analysis tools can query evidence through
`EvidenceView.query_events(...)`:

```python
failed_observations = evidence_view.query_events("observation", status="failed")
recent_engine_events = evidence_view.query_events(
    ["controller_decision", "engine_snapshot"],
    engine_id="llm_engine",
    limit=20,
)
```

Each result has a normalized envelope containing `event_type`, ids, status,
timestamp when available, and the original `record`. This keeps controller code
from needing to know which JSONL file an event came from.

The alpha scheduler supports an explicit retry policy:

```yaml
execution:
  scheduler:
    implementation: builtin.local_scheduler
    config:
      retryPolicy:
        maxAttempts: 2
        retryStatuses: [failed, timeout]
        delaySeconds: 0
```

Retries are attempt-level, not optimizer-level. OptPilot may record failed
attempt observations for auditability, but the scheduler returns only the final
attempt to the controller and engine. This keeps user-owned optimization
algorithms focused on candidate outcomes while preserving the evidence needed
to debug flaky evaluators.

## 21. Policy Model

### 21.1 Access Policy

Access policy defines what a controller or engine may inspect.

Levels:

- `InvocationOnly`
- `SchemaAware`
- `TraceAware`
- `CodeAwareReadOnly`
- `FullStudyContext`

### 21.2 Mutation Policy

Mutation policy defines what a component may change.

Scopes:

- `NoMutation`
- `StudyArtifactOnly`
- `StudyWorkspaceOnly`
- `EngineConfigOnly`
- `ControllerConfigOnly`

The platform should validate and record these policies now, then gradually enforce them as the runtime matures.

## 22. Resource And Sandbox Model

OptPilot should declare resource and sandbox needs, but delegate enforcement to backends and sandbox providers.

Resource profile example:

```yaml
resourceProfile:
  cpu: 4
  memoryGiB: 16
  gpu: 1
  gpuClass: A100
  timeoutSeconds: 3600
```

Sandbox spec example:

```yaml
sandboxSpec:
  runtimeType: container
  networkPolicy: disabled
  readOnlyMounts:
    - /protected/simulator
  writableWorkspace: /workspace
  cleanupPolicy: always
```

OptPilot should record requested and assigned resources. Actual placement should be handled by Docker, Kubernetes, SLURM, Ray, or another backend.

## 23. Evidence Layout

A local run should produce files like:

```text
run_dir/
  study_spec.json
  run_policy.json
  run_lineage.json
  environment_snapshot.json
  controller_decisions.jsonl
  engine_snapshots.jsonl

  summary.json
```

This local layout is a reference implementation. Later evidence stores may use SQLite, object storage, or a service backend, but they should expose the same conceptual entities.
