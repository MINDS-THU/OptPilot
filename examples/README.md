# Examples

`examples/` contains curated, useful OptPilot integrations. The UI scans this folder by default, together with `user_catalog/`.

The main example wraps a strategic-airlift DEVS simulator generated by `devs_gen_code`/`devs_gen_gallery` and evaluates file-edit candidates against it.

## Layout

```text
examples/
  environments/
    strategic_airlift_devs/
      environment.yaml
      evaluator.py
      instances/sa_default.yaml
      prompts/sa_file_edit_system_prompt.md
  methods/
    baseline_file_copy/
      method.yaml
      method.py
    openai_file_editor/
      method.yaml
      method.py
  studies/
    sa_baseline.yaml
    sa_openai_file_editor.yaml
```

Environment and method directories own reusable implementation code plus reusable config variants. Study files are project-centric: each study chooses one environment config, one method config, objective, instances, budget, and runtime policy.

## Strategic Airlift DEVS Environment

The environment config at `examples/environments/strategic_airlift_devs/environment.yaml`:

- copies the generated simulator from `resource/devs_gen_gallery/simulators/SA/simulator`
- exposes `MissionController.py` as the editable file
- runs `examples.environments.strategic_airlift_devs.evaluator:evaluate`
- records simulator events and summary metrics

Before running the example, make sure the generated simulator exists at the configured `workspace.copy.from` path.

## Studies

Run the baseline method first to verify that the simulator can be copied and evaluated:

```bash
uv run optpilot run examples/studies/sa_baseline.yaml
```

Run the OpenAI-compatible file-editing method after setting an API key:

```bash
export OPENROUTER_API_KEY=...
uv run optpilot run examples/studies/sa_openai_file_editor.yaml
```

## User Catalog

Put your own integrations under `user_catalog/` using the same layout: environment directories own environment code/configs, method directories own method code/configs, and studies bind them together.
